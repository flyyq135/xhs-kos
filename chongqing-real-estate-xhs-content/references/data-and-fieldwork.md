# 数据与实地研判要求

## 适用场景

写具体楼盘、小区、板块研判时必须使用本文件。目标是避免产出常识化内容，把笔记升级为“数据支撑 + 实地体验 + 信息差判断 + 看房动作”。

## 四类证据

### 1. 价格与成交

优先收集：

- 当前挂牌区间
- 近90天成交样本
- 近半年成交样本
- 同户型、同面积段、同楼栋价差
- 成交周期
- 当前在售房源数量
- 价格口径：建面、套内、总价、单价、装修、税费是否同口径

拿不到实时数据时，不要编。内部底稿写成：

```text
价格和成交需以当日贝壳/链家/房管口径核实。当前内容先给判断框架，不把价格写成确定结论。
```

对外笔记不要直接写“数据来自某平台链接”。改成经纪人视角：

```text
看这个小区，不要只问均价。要把同面积段、同楼栋位置和近阶段成交口径拆开看。
```

正式笔记要露出关键数据，但不露出来源链接。优先选择客户能快速理解的 2 至 4 个数据锚点：

- 组团/分期楼龄跨度：例如“有的组团十多年楼龄，有的组团更新，不能混着看。”
- 参考单价或总价差：例如“同小区不同组团，参考单价可能差出一截，先拆组团再谈价格。”
- 挂牌/在售量级：例如“在售量大时，更要看同户型真实成交，而不是只看挂牌。”
- 近90天/近半年成交口径：例如“近期同户型有没有真实成交，比挂牌价更能说明问题。”
- 步行/换乘距离：例如“换乘距离不是问题，真正要实走的是楼栋门口到站厅这段。”
- 户型面积段：例如“同样两房/三房，面积段和户型效率会直接影响总价和居住感。”
- 梯户比/车位/出入口：例如“改善客户要把电梯等待、车位和出入口放进价格判断里。”

禁止只写“数据需核实”而不给任何数据锚点。若公开数据不足，至少提供“应该核实哪三个数字”的清单，并把结论写得更克制。

### 2. 空间与通勤

优先收集：

- 小区到最近轨道站的真实步行路径
- 是否有坡坎、绕行、天桥、地下通道、红绿灯
- 早晚高峰是否依赖过桥、隧道或主干道
- 车库出入口是否容易拥堵
- 到核心商圈、学校、医院、公园的实际动线
- 雨天和夜间步行体验

不要只写“交通方便”。改成：

```text
工作日晚高峰从楼栋门口走到轨道站，重点看坡坎、红绿灯、人流、换乘和耗时。
```

### 3. 产品与楼栋

优先收集：

- 分期和楼栋分布
- 楼龄和交付时间
- 户型面积段
- 梯户比
- 朝向、楼层、采光、视野
- 临街、临商业、临轨道、临车库出入口
- 中庭、边户、端头、底商影响
- 物业、车位、停车动线和小区维护状态

同一个小区不能只给一个结论。必须说明：

```text
这个小区的判断不能脱离楼栋和户型。临街楼栋、靠中庭楼栋、靠商业楼栋的体验可能完全不同。
```

### 4. 生活与实地体感

优先收集：

- 白天和夜间噪音
- 商业人流、车流和外卖骑手动线
- 小区出入口是否拥堵
- 底商油烟、灯光、广场舞、夜间经营
- 电梯等待、楼道维护、垃圾点、快递点
- 带娃、老人、宠物和停车的日常便利

不要只写“商业成熟”。改成：

```text
商业近是优势，但要看楼栋是否受车流、人流、夜间灯光、底商油烟和停车影响。
```

## 数据采集表

交付具体小区笔记时，至少附一个内部简表。这个表给经纪人复核和备答使用，不默认放进小红书正文：

| 类别 | 字段 | 当前状态 |
|---|---|---|
| 价格成交 | 挂牌区间、近90天成交、成交周期 | 已核实 / 待核实 |
| 产品楼栋 | 分期、楼龄、户型面积段、梯户比 | 已核实 / 待核实 |
| 位置动线 | 到轨道真实步行、坡坎、红绿灯、换乘 | 已核实 / 待核实 |
| 居住干扰 | 临街、临商业、噪音、车库出入口 | 已核实 / 待核实 |
| 教育规划 | 学校、规划、划片、政策年份 | 已核实 / 待核实 |

## 台前/台后表达规则

### 台后必须保留

- 数据平台或资料来源
- 采样日期
- 价格、成交、挂牌、供应、政策等口径
- 哪些数据已核实，哪些需要现场或系统二次核实
- 由数据推导出的判断

### 台前默认不展示

- 来源链接列表
- 平台截图式堆砌
- “我查了某某平台”这类过程描述
- 过多数字表格
- 未经解释的专业名词

### 台前要转化成

- 一句专业判断
- 一个客户听得懂的变量
- 2 至 4 个自然出现的数据锚点
- 一个现场核实动作
- 一个适合/不适合人群
- 一个风险边界

示例：

```text
内部底稿：A平台显示近90天成交样本偏少，挂牌房源多集中在某面积段，需核实同户型成交。
对外表达：这个小区不能只看挂牌价。比如同面积段房源如果挂牌不少，但近阶段同户型真实成交不多，议价空间和客户接受度就要分开判断。
```

```text
内部底稿：不同组团建成年代跨度较大，参考单价不在同一水平。
对外表达：同样叫二十四城，有些组团已经是十多年楼龄，有些产品更新，参考价也不是一个层级。所以我不会用一个均价概括整个小区。
```

## 实地踩盘清单

至少给用户 5 个现场动作：

1. 工作日晚高峰从目标楼栋走到最近轨道站。
2. 晚上 8 点后看临街、临商业、车库出入口和中庭噪音。
3. 从车库开到主干道，观察排队、转弯和拥堵。
4. 同小区比较临街楼栋、中庭楼栋和靠商业楼栋。
5. 看电梯、楼道、快递点、垃圾点和儿童活动区维护。
6. 雨天或下班后再看一次坡坎、路面和照明。
7. 用同预算对比 2 个替代小区或板块。

## 实地体验语言库

用具体动作替代抽象词：

| 不要只写 | 改成 |
|---|---|
| 交通方便 | 从楼栋门口到轨道站，实测坡坎、红绿灯、人流和换乘距离 |
| 商业成熟 | 商业近是优势，但要看车流、人流、夜间灯光和停车影响 |
| 配套完善 | 分别看日常买菜、带娃活动、老人就医和晚间回家动线 |
| 居住舒适 | 看楼栋是否临街、采光是否被遮挡、电梯等待和中庭噪音 |
| 适合改善 | 先看面积段、梯户比、车位、电梯、安静度和老人小孩动线 |
| 有学校 | 以当年官方招生政策为准，核实户籍、产权、实际居住等要求 |

## 经纪人专业表达模板

用“我怎么帮客户判断”替代“资料显示”：

```text
我看这个小区，通常先拆三个变量：楼栋位置、同面积段成交口径、回家动线。
```

```text
如果只看商圈，这个小区很容易被高估；如果把楼栋和噪音拆开看，反而能筛出更适合自住的房源。
```

```text
带客户看这里，我不会第一时间聊均价，而是先让他走一遍晚高峰回家路线。
```

```text
这类成熟商圈房，优势不用多说，真正要帮客户把关的是：便利有没有变成干扰。
```

## 差异化评分

正式交付前自评 20 分制，低于 14 分不得交付，必须补数据或重写角度。

| 项目 | 分值 |
|---|---:|
| 有明确反常识或信息差判断 | 3 |
| 至少 3 个数据点或数据口径，且正文露出 2 个以上数据锚点 | 3 |
| 至少 2 个实地观察或现场动作 | 3 |
| 写出楼栋/分期/户型差异 | 3 |
| 写出适合与不适合人群 | 2 |
| 写出风险边界和待核实项 | 2 |
| 给出具体看房清单 | 2 |
| 合规，无收益/入学/成交承诺 | 2 |

额外检查：正式笔记如果出现来源链接列表、平台出处堆砌、未经转化的数据表，需改成内部核验表或经纪人判断表达。

如果正式笔记没有具体数字、范围、量级或时间口径，即使内部有数据，也只能按“数据点”得 1 分。

## 不合格内容信号

出现以下情况，重写：

- 只是在介绍位置、商圈、轨道和学校。
- 没有任何数据口径或待核实口径。
- 没有楼栋、分期、户型、动线或噪音差异。
- 看房建议只有“多看看、问清楚”。
- 标题只有“值得买吗”“适合谁”“优缺点分析”。
- 把教育、成交、价格或规划写成确定承诺。
